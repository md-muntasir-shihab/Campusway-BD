Website Intelligence Report: CampusWay
Audit Date: 2026-04-14
Audit Scope: Repository-only (frontend, frontend-next, backend, docs). No live URL/session logs/screenshots were provided in this pass.
Evidence Legend: Observed = directly visible in repository files. Inferred = derived from route wiring, naming, model shape, or workflow documentation.

1. WEBSITE IDENTITY
- Website name: CampusWay
- Website purpose: Multi-surface education platform combining public discovery, student exam operations, and admin operations.
- Target audience:
  - Public visitors looking for universities, resources, notices/news, subscription plans.
  - Students using exam, result, payment, profile, and support flows.
  - Admin/staff running academic content, exam operations, subscriptions, finance, communication, and security settings.
- Platform type: Role-based web platform (public site + student portal + admin control plane).
- Business model:
  - Subscription plans and manual payment handling are core.
  - Secondary monetization/retention through exam ecosystem and recurring student engagement.
- Core objective:
  - Act as a centralized admission/exam readiness platform with structured admin governance.

2. TECHNICAL PROFILE
- Frontend stack (Observed):
  - Primary app: Vite + React + React Router + React Query + Tailwind.
  - Secondary app: Next.js hybrid app with smaller route coverage.
  - Auth client behavior: JWT access token bootstrap, refresh flow via backend, EventSource session checks.
- Backend stack (Observed):
  - Express + TypeScript API.
  - MongoDB via Mongoose models.
  - Middleware: helmet, hpp, mongo sanitize, request sanitizer, rate limiting, role guards.
- Database type (Observed): MongoDB document model with large multi-domain schema inventory.
- Hosting/deployment clues (Observed):
  - Frontend hosted on Firebase.
  - Backend targeted to Azure App Service.
  - Dockerfile exists for backend.
- UI framework clues (Observed):
  - Tailwind utility styling and custom component composition.
  - Framer Motion used in exam runner interactions.
- API style clues (Observed):
  - REST-style JSON endpoints grouped by audience: public, student, admin, webhooks.
  - Secret admin path and legacy alias paths both supported.
- Real-time/async/sync clues (Observed):
  - SSE used for auth/session and admin user stream.
  - Cron jobs for exams, news, subscriptions, notifications, retention, finance recurring.
  - Async import/export and workflow jobs exist in multiple domains.

5. USER ROLES AND PERMISSIONS
Superadmin
- Allowed: all modules and all actions in matrix.
- Restricted: effectively none.
- Visibility: full system.
- Risk: single point of complete power.

Admin
- Allowed: broad module access including publish/approve/export/bulk; some delete scopes.
- Restricted: fewer destructive privileges than superadmin.
- Visibility: almost full.
- Risk: high operational impact, needs strong audit discipline.

Moderator
- Allowed: content-heavy modules and partial support/notifications/report views.
- Restricted: limited destructive and system-level settings.
- Visibility: medium-high.
- Risk: mixed action scopes can create confusion if UI does not explain denied operations.

Editor
- Allowed: news/resources/question-bank and selective home control edits.
- Restricted: finance/security/team high-control actions.
- Visibility: content domain.

Viewer
- Allowed: view-only across admin modules.
- Restricted: no create/edit/delete/publish/approve.
- Visibility: broad read.

Support_agent
- Allowed: support center heavy actions and some report view.
- Restricted: finance/exam/content admin in general.

Finance_agent
- Allowed: payments and finance center operations including bulk/export.
- Restricted: most content and security modules.

Chairman
- Allowed: reports analytics view/export and security logs view.
- Restricted: broad operational controls.
- Visibility: oversight slice only.
- Concern: chairman surface appears present but low evidence of deep feature maturity.

Student
- Allowed: student and public protected flows only.
- Restricted: all admin modules and privileged APIs.

Guest
- Allowed: public pages and anonymous write endpoints.
- Restricted: all protected features.

Suspicious permission leak vectors
- Frontend permission checks rely on permissionsV2 presence; if stale payloads occur, UI can render actions that backend later denies.
- Legacy permission bridge and permissionsV2 coexistence increases mismatch risk during migrations.

6. FEATURE-BY-FEATURE EXPLANATION
Login/logout
- Purpose: controlled role access.
- Flow: client login endpoint by portal type -> token + user -> auth/me verification -> route redirect.
- Backend dependency: auth controllers/session models.
- DB dependency: User, ActiveSession, LoginActivity.
- Edge cases: session invalidated, idle timeout, force logout events.
- Failure points: insecure fallback secrets in production mode warning path; auth bootstrap failures can force silent logout.
- Status: Working with security caveats.

Account creation
- Purpose: student onboarding.
- Flow: register endpoint -> profile baseline -> login/verification path.
- DB: User + StudentProfile.
- Failure points: anti-abuse relies partly on optional App Check.
- Status: Working.

Password change/reset
- Purpose: account recovery/security hygiene.
- Flow: forgot-password -> reset token -> set new password; admin can force resets.
- DB: PasswordReset and user password history fields.
- Failure points: email delivery and token expiry edge handling not fully visible.
- Status: Working.

Subscription purchase/request
- Purpose: monetize and gate exam/support features.
- Flow: plan browse -> checkout/request payment -> admin/finance verification -> user subscription update.
- DB: SubscriptionPlan, UserSubscription, ManualPayment.
- Edge cases: stale subscription snapshot inside user document.
- Failure points: external payment automation not strongly evidenced in frontend.
- Status: Partially automated, operationally working.

Manual approval flow
- Purpose: controlled high-risk actions and payment proofs.
- Flow: create pending actions/payment evidence -> admin approval/rejection.
- DB: ActionApproval, payment-related models.
- Failure points: reviewer throughput and inconsistent context in UI.
- Status: Working but process-heavy.

Payments
- Purpose: recording and reconciling student payments.
- Flow: student request/upload proof -> admin verify -> finance records -> invoice/ledger updates.
- DB: ManualPayment, FinanceTransaction, FinanceInvoice, StudentDueLedger.
- Failure points: proof authenticity checks are weak without OCR/bank verification.
- Status: Working baseline.

Expiry alerts
- Purpose: retention and access gating.
- Flow: subscription cron and notification system generate alerts.
- Failure points: notification provider reliability unknown.
- Status: Working with delivery confidence gaps.

Exam creation
- Purpose: author and schedule assessment products.
- Flow: admin exam form -> question linking -> publish states -> student access.
- DB: Exam + question bank + exam center/import profiles.
- Edge cases: internal vs external delivery mode conflict, many legacy aliases.
- Failure points: field duplication can cause drift in behavior.
- Status: Working, complex.

Exam taking
- Purpose: session-based assessment.
- Flow: start session -> fetch questions -> local+server autosave -> submit manual/timeout.
- DB: ExamSession with answer history and anti-cheat fields.
- Edge cases: offline mode, queue flush timing, timeout auto-submit failure.
- Failure points: local storage corruption, large answer state, unstable network.
- Status: Working with high operational complexity.

Question bank usage
- Purpose: reusable question assets and governance.
- Flow: create/import/revise/approve questions -> attach to exams.
- DB: Question, QuestionBankQuestion, set/usage/revision/media models.
- Failure points: dual legacy/new model pathways.
- Status: Working with technical debt.

Banner upload
- Purpose: controlled media publishing.
- Flow: upload -> secure upload registration -> banner manager references.
- DB: SecureUpload and Banner.
- Failure points: file type trust still partially extension-based.
- Status: Working.

Live alerts and notifications
- Purpose: timely user/admin communication.
- Flow: notification jobs/templates/providers -> user inbox and optional channel delivery.
- DB: Notification, NotificationTemplate, NotificationProvider, NotificationJob, NotificationDeliveryLog.
- Failure points: campaign/provider runtime verification documented as pending.
- Status: Partially validated.

Finance dashboards
- Purpose: operational and executive finance visibility.
- Flow: transactional models -> dashboard queries -> charts/exports.
- Failure points: heavy query surfaces, export/import stress.
- Status: Active but runtime quality uneven.

Export features
- Purpose: reporting and operational transfer.
- Flow: module exports to files (results, students, finance, news, etc).
- Failure points: broad export surfaces increase data leak and performance risk.
- Status: Working with control risk.

Support tickets
- Purpose: student issue handling.
- Flow: create ticket -> thread replies -> admin support processing.
- Failure points: SLA/escalation mechanics not clearly visible.
- Status: Working baseline.

Audit logs
- Purpose: trace privileged actions and workflow changes.
- Flow: middleware/service events -> audit collections.
- Failure points: not all sensitive flows guarantee uniform logging depth.
- Status: Working but consistency unclear.

Notifications inbox behavior
- Purpose: user-facing visibility of system messages.
- Flow: delivery records -> student/admin read models.
- Failure points: duplicates or delayed states possible if jobs/providers lag.
- Status: Working.

Image upload/link handling
- Purpose: media management in news/banner/resource/profile contexts.
- Flow: upload or URL -> secure upload metadata -> served via /uploads.
- Failure points: URL-based content trust and legacy orphan files.
- Status: Working with review risk.

7. DATA MODEL / INFORMATION FLOW
Input to output data flow
- User input:
  - Public forms (contact/help feedback/events/share tracking).
  - Student forms (profile, support, exam answers, subscription requests).
  - Admin forms (exams, question bank, news moderation, settings, finance records).
- Admin action:
  - Approval, publish, schedule, verification, role/permission control.
- Backend processing:
  - Validation, role/action checks, dedupe/normalization, cron orchestration.
- Database save:
  - Multi-collection writes by domain models.
- Frontend rendering:
  - React Query-driven fetches and role-conditional shells.
- Notification/update flow:
  - notification jobs/providers/delivery logs + SSE for some live events.

Main collections by domain (Observed)
- Identity/access: User, StudentProfile, AdminProfile, ActiveSession, SecurityToken, PasswordReset, OtpVerification, TeamRole, MemberPermissionOverride.
- Content/public: HomeSettings, WebsiteSettings, Banner, ContentBlock, Resource, HelpArticle.
- Universities: University, UniversityCategory, UniversityCluster.
- News: News, NewsSource, NewsFetchJob, NewsAuditEvent, NewsSystemSettings.
- Exams: Exam, ExamSession, ExamResult, QuestionBankQuestion, QuestionRevision, ExamImport* models.
- Subscription/finance: SubscriptionPlan, UserSubscription, ManualPayment, FinanceTransaction, FinanceInvoice, FinanceBudget, FinanceRecurringRule, StudentDueLedger.
- Communication/support: Notification*, ContactMessage, SupportTicket, SupportTicketMessage.
- Ops/security: AuditLog, SecurityAlertLog, SecurityRateLimitEvent, ImportExportLog, BackupJob, SecureUpload.

Key relationships and dependencies (Inferred from models and routes)
- User -> UserSubscription and denormalized user.subscription cache.
- Exam -> ExamSession -> ExamResult with optional profile sync/import overlays.
- NewsSource -> News items via ingestion workflows.
- Notification templates/providers -> Notification jobs -> delivery logs -> read receipts.
- StudentGroup memberships influence exam audience and communication targeting.

Repeated/duplicated fields and likely stale patterns
- Exam includes both legacy and newer naming for access/subscription/payment controls.
- User has both legacy permissions and permissionsV2.
- Model folder contains legacy lowercase model files and active PascalCase models.
- Route layer keeps many legacy aliases that preserve behavior but increase drift risk.

Likely foreign keys/references
- ObjectId references across user, plan, exam, center, group, role, and audit entities.
- Many denormalized snapshots (for performance/history) imply sync risk when source changes.

8. UI / UX STRUCTURE
Navigation structure
- Public: navbar + footer + route-level SEO.
- Student: protected layout with dashboard-centric shortcuts.
- Admin: guarded shell with module tabs and nested route trees.

Layout patterns
- Card-heavy surfaces for student dashboard and public pages.
- Table-heavy surfaces for admin operations (students, finance, imports).
- Modal/overlay usage in profile popup, confirmations, exam runtime prompts.

Responsive behavior
- Tailwind breakpoints are present across major pages.
- Public and student pages are generally responsive.
- Admin data grids are usable but clearly desktop-first.

Loading, empty, error states
- Loading skeletons and spinner use are present in several key pages.
- Empty and error states exist but depth/consistency varies by module.
- Save and submit errors surfaced with toasts in runtime exam flow.

Consistency issues
- Mixed style token usage (hardcoded shades plus semantic tokens).
- Repeated route aliases and duplicated pages add UX confusion.
- Permission-denied UX is blunt (access-denied redirects) with limited guidance.

Design quality level
- Mid-level product UX with good functional coverage.
- Not yet a fully unified design system experience across all admin modules.
